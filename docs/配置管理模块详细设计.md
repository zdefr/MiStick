## 2. 模块概述

### 2.1 职责

配置管理模块负责应用配置的存储、读取、更新、验证和备份。配置文件保持用户可读可改，登录态相关敏感内容不明文落在配置文件中；当前首期默认只保存 `mijia-api` 认证文件路径和登录元信息。

### 2.2 首期范围

**首期必须实现：**
- 启动时加载配置
- 配置校验与默认值补齐
- 配置项读写
- 配置备份与恢复
- 管理 `miHome.authStoragePath` 与服务配置
- 配置文件权限收敛到当前用户可读写

**首期不做：**
- OpenClaw 配置管理
- 批处理脚本配置管理
- 多账号切换策略
- 直接解析或改写 `mijia-api` 认证文件内容

### 2.3 设计边界

- 配置模块负责 `config.json`。
- `mijia-api` 的认证文件由 MiHome Bridge Service 维护。
- 配置模块只保存认证文件路径、账号 ID、区域、最后登录时间等元信息。
- 若存在历史遗留的 `miHome.token` 字段，仅作为兼容迁移字段处理，默认新配置不再写入它。

## 3. 配置结构

### 3.1 配置文件结构

```typescript
interface AppConfig {
  version: string;
  userId: string;
  miHome: {
    provider: 'mijia-api';
    accountId?: string;
    authStoragePath: string;
    region: 'cn' | 'de' | 'us';
    lastLoginAt?: string;
    token?: string;              // 兼容迁移字段，若存在则加密存储
  };
  services: {
    mihomeBridge: {
      baseUrl: string;
      timeoutMs: number;
    };
    localControl: {
      enabled: boolean;
      baseUrl: string;
      timeoutMs: number;
    };
  };
  window: {
    width: number;
    height: number;
    x?: number;
    y?: number;
    alwaysOnTop: boolean;
    opacity: number;             // 0.1 - 1.0
    skipTaskbar: boolean;
  };
  appearance: {
    theme: 'light' | 'dark' | 'system';
    fontSize: number;
    language: 'zh-CN' | 'en';
  };
  devices: {
    autoRefresh: boolean;
    refreshInterval: number;     // 秒
    lastSyncAt?: string;
  };
  logging: {
    level: 'debug' | 'info' | 'warn' | 'error';
    maxFiles: number;
    maxSize: string;
  };
  createdAt: string;
  updatedAt: string;
}
```

说明：
- 配置文件是面向用户的可编辑文件，非敏感字段保持明文。
- `miHome.authStoragePath` 保存的是认证文件路径，不是认证内容。
- `miHome.token` 不再是首期主路径，仅作为兼容字段保留。

### 4.2 核心类说明

#### 4.2.1 ConfigManager

```typescript
class ConfigManager {
  private configPath = path.join(app.getPath('userData'), 'config.json');
  private backupDir = path.join(app.getPath('userData'), 'backups');
  private config: AppConfig | null = null;
  private cryptoService = new CryptoService();

  async load(): Promise<AppConfig> {
    const raw = await fs.readFile(this.configPath, 'utf-8');
    const config = JSON.parse(raw) as AppConfig;

    if (config.miHome.token) {
      config.miHome.token = await this.cryptoService.decryptLegacyToken(config.miHome.token);
    }

    this.validate(config);
    this.config = config;
    return config;
  }

  async save(patch?: Partial<AppConfig>): Promise<void> {
    const nextConfig = mergeConfig(this.config ?? this.getDefaultConfig(), patch ?? {});
    this.validate(nextConfig);

    const toSave = structuredClone(nextConfig);
    if (toSave.miHome.token) {
      toSave.miHome.token = await this.cryptoService.encryptLegacyToken(toSave.miHome.token);
    }

    toSave.updatedAt = new Date().toISOString();
    await this.ensureBackupBeforeWrite();
    await fs.writeFile(this.configPath, JSON.stringify(toSave, null, 2), 'utf-8');
    await this.ensureFilePermission();
    this.config = nextConfig;
  }
}
```

#### 4.2.2 CryptoService

```typescript
class CryptoService {
  private algorithm = 'aes-256-gcm';
  private keyProvider = new KeyProvider();

  async encryptLegacyToken(plainText: string): Promise<string> {
    const key = await this.keyProvider.getOrCreateKey();
    const iv = crypto.randomBytes(12);
    const cipher = crypto.createCipheriv(this.algorithm, key, iv);
    const encrypted = Buffer.concat([cipher.update(plainText, 'utf8'), cipher.final()]);
    const authTag = cipher.getAuthTag();
    return Buffer.concat([iv, authTag, encrypted]).toString('base64');
  }

  async decryptLegacyToken(cipherText: string): Promise<string> {
    const key = await this.keyProvider.getOrCreateKey();
    const payload = Buffer.from(cipherText, 'base64');
    const iv = payload.subarray(0, 12);
    const authTag = payload.subarray(12, 28);
    const encrypted = payload.subarray(28);
    const decipher = crypto.createDecipheriv(this.algorithm, key, iv);
    decipher.setAuthTag(authTag);
    return Buffer.concat([decipher.update(encrypted), decipher.final()]).toString('utf8');
  }
}
```

说明：
- 配置文件权限与登录态保护是两件不同的事。
- 当前首期的真实登录态主要存在 `mijia-api` 认证文件中，而不是 `config.json`。
- 兼容字段 `miHome.token` 若存在，仍按加密字段处理。

## 6. 流程设计

### 6.1 配置加载流程

```text
应用启动
  ↓
ConfigManager.load()
  ↓
读取 config.json
  ↓
存在 miHome.token ?
  ├─ 否：直接校验配置
  └─ 是：解密兼容 token 后再校验配置
  ↓
返回内存态配置对象
```

### 6.2 配置更新流程

```text
用户修改配置
  ↓
ConfigManager.save(patch)
  ↓
合并默认值与 patch
  ↓
校验字段合法性
  ↓
仅对兼容字段 miHome.token 加密
  ↓
写入配置文件
  ↓
收敛文件权限
```

### 6.3 认证文件协作流程

```text
M01 登录成功
  ↓
MiHome Bridge Service 写入 mihome-auth.json
  ↓
M01 更新 config.miHome.authStoragePath / accountId / lastLoginAt
  ↓
M04 保存配置
```

### 7.2 配置文件权限

```typescript
async function ensureFilePermission(configPath: string): Promise<void> {
  // 目标：配置文件仅允许当前用户读写。
  // 在支持 POSIX 权限位的环境可使用 chmod；
  // 在 Windows 环境中由实现层映射为等价权限控制。
  await chmod(configPath, 0o600);
}
```

说明：
- 权限控制针对整个配置文件。
- 配置文件仍允许当前用户手工修改。
- `mihome-auth.json` 的权限收敛由 MiHome Bridge Service 负责，不由 M04 直接管理内容。

## 8. 安全设计

### 8.1 敏感信息保护范围

当前首期保护的敏感信息包括：
- `mihome-auth.json` 认证文件内容
- 历史兼容字段 `miHome.token`（若存在）

以下字段保持明文，便于用户查看和手工编辑：
- `miHome.authStoragePath`
- `miHome.region`
- `services.*`
- `window.*`
- `appearance.*`
- `devices.*`
- `logging.*`

### 8.2 密钥管理原则

- 加密密钥由 `KeyProvider` 统一提供，不散落在业务代码中。
- 解密失败时，不回退到明文逻辑，直接抛出 `DecryptFailed`。
- 业务层只能调用 `encryptLegacyToken/decryptLegacyToken`，不直接操作算法细节。

### 8.3 配置校验规则

| 字段 | 规则 |
|------|------|
| `window.opacity` | 0.1 - 1.0 |
| `window.width` | >= 320 |
| `window.height` | >= 400 |
| `devices.refreshInterval` | 30 - 3600 秒 |
| `appearance.fontSize` | 12 - 24 |
| `services.mihomeBridge.baseUrl` | 必须为有效 URL |
| `services.localControl.baseUrl` | 启用时必须为有效 URL |

## 9. 异常处理

### 9.1 异常类型

| 异常 | 触发条件 | 处理方式 |
|------|----------|----------|
| ConfigNotFound | 配置文件不存在 | 创建默认配置 |
| ConfigCorrupted | JSON 解析失败或结构非法 | 恢复备份或重建默认配置 |
| DecryptFailed | 兼容 token 解密失败 | 清空旧字段并提示重新登录 |
| ValidationFailed | 字段不符合规则 | 拒绝写入并提示用户 |
| WriteFailed | 文件写入失败 | 重试并记录日志 |
| PermissionFailed | 权限设置失败 | 记录警告并继续运行 |
