# 第三方依赖梳理

## 1. 文档目的

本文档用于梳理当前首期方案涉及的第三方依赖，包括：
- 运行时依赖
- 构建与打包依赖
- 测试依赖
- 外部服务依赖
- 暂未冻结、需要后续确认的候选依赖

说明：
- 本文档以当前设计为基础，关注“依赖角色”和“是否需要引入”，不强行锁死所有版本号。
- 版本号建议以开工时的稳定版为准。

---

## 2. 依赖分层总览

| 类别 | 说明 | 当前状态 |
|------|------|----------|
| 桌面端运行时依赖 | Electron 应用、前端 UI、状态管理 | 必需 |
| 本地持久化依赖 | 配置、SQLite、日志 | 必需 |

| OpenClaw 相关依赖 | Agent / 脚本能力 | 首期不引入 |

## 5. 外部服务与非包依赖

### 5.1 米家云端服务

| 依赖 | 类型 | 用途 | 是否必需 | 风险等级 |
|------|------|------|----------|----------|
| 米家账号体系 | 外部服务 | 登录与登录态保持 | 是 | 高 |
| 米家云端设备接口 | 外部服务 | 同步家庭、房间、设备列表 | 是 | 高 |

说明：
- 这是首期方案中最关键的外部依赖。
- 当前文档只定义能力抽象，未在代码层锁死具体 SDK。
- 开发前需要先通过 Spike 验证最低闭环可用性。

调研结论：
- 小米账号 OAuth2 接入路径是公开可查的，桌面客户端可参考官方 OAuth2 文档接入。
- 官方文档明确支持 `Authorization Code` 和 `Implicit Grant` 两种模式；对本项目这种需要长期登录态与刷新能力的桌面应用，更建议采用 `Authorization Code`。
- 官方公开资料中可以确认存在与“智能家庭服务”相关的 scope 权限，但没有查到可直接自助接入“家庭/房间/设备列表”的完整公开 API 文档。
- 因此，`米家账号体系` 可以按官方账号开放平台接入；`米家云端设备同步能力` 目前不能假设仅凭公开文档即可直接落地，仍需做权限申请与能力验证。
- 结合当前项目实际推进情况，个人开发者路径下官方应用接入无法作为可落地前提，因此本项目**不再把官方账号体系/智能家庭权限当作首期可依赖基座**。

接入建议：
- 账号登录：采用 OAuth2 `Authorization Code` 流程，由受控服务端换取和刷新 Token，桌面端保存必要登录态。
- 云端同步：先申请智能家庭相关权限，再确认是否能获得正式接口文档或合作接入说明；在此之前，业务代码只实现 `Cloud Sync Adapter` 抽象，不写死具体 HTTP API。

当前项目决策：
- 官方路线降级为“背景调研路线”，不作为首期实施前提。
- 当前仓库 demo 已验证 `mijia-api` 可完成扫码登录、会话复用、设备同步和基础云端控制。
- 首期云端登录/同步/控制默认走 `mijia-api`。
- 文档中的 `Cloud Sync Adapter` 保留抽象层，但默认实现优先考虑社区方案而非官方开放平台。

官方来源链接：
- OAuth2 总览：https://dev.mi.com/docs/passport/oauth2/
- Authorization Code 授权接口：https://dev.mi.com/docs/passport/authorization-code/
- Refresh Token 刷新接口：https://dev.mi.com/docs/passport/refresh-token/
- 小米帐号开放 API：https://dev.mi.com/docs/passport/open-api/
- Scope 权限列表：https://dev.mi.com/docs/passport/scopes/

## 6. 待冻结的候选依赖

这些依赖在当前方案里需要“保留位置”，但还不建议现在就写死：

### 6.1 云端接入候选依赖

| 候选依赖 | 用途 | 当前状态 | 说明 |
|----------|------|----------|------|
| 官方 OAuth2 + 智能家庭相关权限 | 账号登录、理论上的云端家庭/设备同步 | 当前项目不依赖 | 作为背景调研保留，但不再作为首期落地前提 |
| `mijia-api` | 云端登录、家庭/设备列表、属性读写、动作执行 | 当前首期默认云端实现 | 社区库，能力完整度高，但非官方且许可证风险高 |

#### 6.1.1 `mijia-api` 方案说明

适用场景：
- 已验证“扫码登录 → 拉家庭/房间/设备 → 云端属性控制/动作执行”最小闭环。
- 当前作为首期默认云端实现，而不再只是 PoC 候选。

已知能力：
- 支持扫码登录，并保存认证信息复用。
- 支持获取家庭列表与设备列表。
- 支持通过设备属性接口读写设备属性。
- 支持执行设备动作和场景。

优势：
- 云端闭环能力集中，适合快速验证首期主链路。
- 比单纯本地协议方案更接近“米家账号登录 + 云端同步”的产品叙述。

风险：
- 非官方库，长期稳定性依赖上游维护和米家接口变化。
- README 与 FAQ 暗示设备支持存在“需要抓包补充”的情况，不适合作为无验证前提的正式承诺。
- GPL-3.0 许可证对闭源/商用项目有明显风险，需要单独评估。

建议定位：
- 当前最适合作为 `Cloud Sync Adapter` 与首期云控能力的默认实现。
- 后续重点从“是否可用”转向“稳定性、许可证和设备覆盖率”验证。
- 在未完成许可证评估和真实设备验证前，不建议直接承诺为长期稳定生产方案。

来源：
- 仓库主页：https://github.com/Do1e/mijia-api
- FAQ：https://github.com/Do1e/mijia-api/blob/main/FAQ.md

### 6.2 本地控制候选依赖

| 候选依赖 | 用途 | 当前状态 | 说明 |
|----------|------|----------|------|
| `python-miio` | 基于本地协议控制设备、查询状态 | 建议优先验证 | 社区中最成熟的 Xiaomi / miIO / MIoT 本地控制库之一 |
| 其他设备协议实现 | 替代 `python-miio` | 可替代 | 若 `python-miio` 对目标设备支持不足，可继续评估 |

#### 6.2.1 `python-miio` 方案说明

适用场景：
- 在局域网内，通过设备 `IP + token` 查询状态、下发控制指令。
- 为本项目的 `control-service` 提供首期本地控制实现。

已知能力：
- 支持大量 Xiaomi / miIO 设备。
- 对现代 MIoT 设备提供 `genericmiot` 路径。
- 支持设备状态查询、属性读取、动作调用等能力。
- 支持通过 `miiocli cloud` 辅助获取云端相关信息和设备上下文。

优势：
- 文档和社区成熟度相对更高。
- 很适合放到 Python 本地控制服务里，和 Electron 主进程解耦。
- 与当前 HLD 的“本地控制适配层”设计天然一致。

风险：
- 非官方库，不保证覆盖所有米家设备。
- 能否控制取决于设备是否支持本地协议，以及能否拿到 `IP + token`。
- 即使设备已在云端同步成功，也不代表一定能通过 `python-miio` 本地控制。

建议定位：
- 作为 M02 首期最优先验证的本地控制候选依赖。
- 先用样本设备验证“灯 / 插座 / 开关”这类最小可控集合，再决定是否作为正式实现基座。

来源：
- 仓库主页：https://github.com/rytilahti/python-miio
- 文档首页：https://python-miio.readthedocs.io/en/stable/index.html

## 10. 风险最高的依赖项

以下依赖最值得在开工早期重点验证：

| 依赖 | 风险原因 | 建议 |

---

## 11. 最终结论

当前第三方依赖结构是健康的，原因在于：
- 首期依赖集合相对克制，没有引入不必要的大型扩展能力。
- UI、配置、缓存、测试这些部分都可以使用成熟通用依赖直接落地。
- 最大风险集中在少数几个外部依赖点，便于集中验证和隔离。

最关键的执行原则是：
- **先冻结最小依赖集合，再验证高风险社区依赖。**
- **不要在 `mijia-api` 和设备协议库验证完成前，过早扩展首期范围。**
